[PLAN] TIPO DE NEGOCIO + CATEGORIAS DINAMICAS (MODELO SIMPLE)

[ESTADO]
Este plan reemplaza el esquema anterior por un modelo mas simple:
- 1 coleccion: configuraciones
- 1 documento: catalogo_negocios
- Ahi viven tipos de negocio + categorias + version

[OBJETIVO]
1) En registro: select de tipo de negocio.
2) Opcion "Otra categoria" + input custom (max 30).
3) Validacion en frontend y backend.
4) Catalogo centralizado en Firebase en 1 solo doc.
5) Cache en IndexedDB para evitar leer Firebase en cada inicio.

Nota:
- "SERVER_TIMESTAMP" es conceptual. En consola Firestore usa timestamp de servidor.

[QUE ES INVALIDACION POR VERSION]
- El frontend guarda en IndexedDB:
  - catalogo (tipos + categorias)
  - version local
- Al iniciar:
  1) lee cache local y muestra rapido
  2) consulta version en Firebase
  3) si version remota != version local, refresca catalogo y pisa cache

Ejemplo:
- local version = 1
- Firebase version = 2
- accion: actualizar cache local con los datos nuevos y guardar version = 2

[QUIEN CREA/EDITA CATEGORIAS]
- Vos (administrador del sistema), no los usuarios finales comunes.
- Opciones:
  1) Editar manualmente en Firebase Console (rapido al inicio).
  2) Script seed (mas prolijo para repetir en otros entornos).
  3) Futuro panel admin interno (si queres autogestion).

[FASE 5 - USUARIOS ACTIVOS (YA EXISTENTES)]
Problema:
- Usuarios/tenants actuales no tienen businessTypeId.

Estrategia recomendada:
1) Compatibilidad inmediata:
  - Si falta businessTypeId, usar fallback "kiosco" en frontend y backend.
2) Migracion gradual:
  - Actualizar tenants existentes con businessTypeId real.
  - Opciones:
    a) Manual por Firebase Console (pocos tenants).
    b) Script de migracion (muchos tenants).
3) Cuando todos tengan dato, mantener fallback solo como seguridad.

Campos a agregar en tenants actuales:
- businessTypeId: "kiosco" | "almacen" | "ropa" | "custom"
- businessTypeLabel: "Kiosco" | "Almacen" | "Tienda de Ropa" | texto custom

Campos a agregar en usuarios empleador (opcionales pero recomendados):
- businessTypeId
- businessTypeLabel

[CRITERIOS DE ACEPTACION]
1) Registro con tipo normal funciona.
2) Registro con custom funciona solo si texto <= 30.
3) Validaciones fallan en frontend y backend para custom invalido.
4) Productos usan categorias del tipo de negocio correcto.
5) Al reiniciar app, usa cache local y refresca solo si cambia version.
6) Usuarios existentes siguen funcionando con fallback mientras migras.
